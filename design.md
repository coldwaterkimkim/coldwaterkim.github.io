# design.md

이 문서는 coldwaterkim.com의 공개 사이트 감성을 고정하기 위한 기준이다.

새 공개 페이지, 홈 개편, 글 목록/상세, 방명록, 미디어 페이지를 만들거나 수정할 때는 먼저 이 문서를 읽고 현재 홈의 감성을 유지한다. 관리자 화면은 사용성이 우선이지만, 공개 사이트에 노출되는 화면은 이 기준을 따른다.

## 디자인 한 문장

1999년에 직접 만든 개인 홈페이지처럼 보이는, 일부러 수제이고 조금 과한 레트로 웹.

핵심은 “현대적으로 다듬은 레트로풍”이 아니라 “옛 웹의 구조와 결함을 지금도 진심으로 쓰는 느낌”이다. 촌스러움은 결함이 아니라 제품 정체성이다.

## 현재 기준 화면

- `index.html`: 공개 사이트의 시각 기준. 새 페이지를 만들 때 가장 먼저 비교할 원본이다.
- `css/styles.css`: 공개 사이트 공통 스타일의 기준.
- `README.md` / `STATUS.md`: 제품 방향과 CMS 운영 방향.
- `admin/`: 관리자 도구. 공개 사이트 감성보다 편집 효율을 우선해도 된다.

## 지켜야 하는 핵심 원칙

1. 진짜 90s 개인 홈페이지 문법을 유지한다.
   - 중앙 정렬된 고정폭 테이블
   - 굵고 단순한 테이블 보더
   - 기본 파란 링크와 보라색 방문 링크
   - 마퀴, 깜빡임, 방문자 카운터, 배지, GIF, WebRing 같은 장식

2. 너무 세련되게 정리하지 않는다.
   - 둥근 카드, 부드러운 그림자, 모던 SaaS 레이아웃, 히어로 섹션을 넣지 않는다.
   - 여백과 정렬을 완벽하게 현대화하지 않는다.
   - “잘 만든 현대 웹”보다 “살아 있는 개인 홈페이지”가 우선이다.

3. 기능은 현대적으로 편해야 하지만 겉모습은 옛 웹이어야 한다.
   - CMS에서 불러온 글, 방명록, 미디어도 테이블/기본 폼/간단한 링크 안에 들어와야 한다.
   - 데이터 로딩, 에러, 빈 상태도 레트로 UI 문법으로 보여준다.

4. 한국어를 중심으로 하고, 영어는 옛 웹 장식처럼 섞는다.
   - 예: `WELCOME`, `UNDER CONSTRUCTION`, `Last Updates`, `Best Viewed`, `Guestbook`
   - 본문 설명과 사용 안내는 한국어가 기본이다.

## 레이아웃 규칙

### 기본 폭

- 공개 사이트는 `760px` 테이블을 기준으로 한다.
- `meta viewport`는 현재처럼 `width=800`을 유지한다.
- 모바일 최적화보다 “800x600에 맞춘 오래된 홈페이지” 느낌이 우선이다.

### 기본 구조

홈처럼 정보가 많은 페이지:

```text
marquee
760px table
  banner row
  sidebar cell + content cell
```

단순 페이지:

```text
760px table
  one content cell
```

복잡한 새 페이지를 만들 때는 홈의 사이드바 구조를 재사용한다. 단순한 글 상세나 About 같은 화면은 한 칸짜리 테이블이어도 괜찮다.

### 테이블 감성

- `table`, `tr`, `td`, `th`, `border="1"`, `cellspacing="0"`, `cellpadding` 같은 오래된 HTML 문법을 숨기지 않는다.
- 리스트형 데이터는 모던 카드 대신 테이블을 기본으로 한다.
- 반복 항목은 카드 그리드보다 `table row`, `dashed box`, `hr`, `ul/li`를 우선한다.

## 색상 기준

현재 색을 기준으로 유지한다.

| 역할 | 값 | 사용 |
| --- | --- | --- |
| 바깥 배경 | `#d8d8d8`, `#e9e9e9` | 대각선 반복 배경 |
| 기본 테이블 | `#ffffff` | 본문 컨테이너 |
| 상단 배너 | `#ffe97a` | 공지, 상태, 카운터 주변 |
| 사이드바 | `#f8f8ff` | 내비게이션 영역 |
| 테이블 헤더 | `#f0f0f0` | 목록 헤더 |
| 노트 텍스트 | `#666` | 작은 안내 |
| 방명록 항목 | `#fcfcf0` | dashed entry |
| 카운터 배경 | `#000000` | 방문자 카운터 |
| 카운터 숫자 | `#00ff00` | 방문자 카운터 |
| 기본 링크 | `#0000EE` | 브라우저 기본 링크 느낌 |
| 방문 링크 | `#551A8B` | 브라우저 기본 방문 링크 |
| 링크 hover | `yellow` | 과한 하이라이트 |

새 색을 추가할 때는 “옛 웹에서 바로 골랐을 법한 원색/기본색”만 쓴다. 파스텔 그라디언트, 투명 유리 효과, 뉴모피즘, 고급스러운 블랙톤은 피한다.

## 타이포그래피

기본:

```css
font-family: "Comic Sans MS", Tahoma, Arial, sans-serif;
```

보조:

- 숫자 카운터, 로그, 코드, 방명록 메타: `"Courier New", monospace`
- 오래된 HTML 느낌이 필요한 곳: `<font>`, `<b>`, `<i>`, `<small>` 사용 가능

주의:

- 현대적인 대형 디스플레이 타이포그래피를 쓰지 않는다.
- 글자 크기 체계는 단순하게 유지한다.
- 헤딩은 예쁘게 정제하기보다 페이지 제목처럼 직설적으로 쓴다.

## 장식 요소

권장:

- `marquee`
- `.blink`
- 방문자 카운터
- 88x31 배지, shields badge
- 작은 GIF

금지에 가까움:

- 모던 히어로 섹션
- 카드형 feature grid

- 복잡한 아이콘 시스템

- smooth한 랜딩페이지 애니메이션
- AI/SaaS 마케팅 카피

## 컴포넌트별 기준

### 내비게이션

- 홈 사이드바의 `NAVIGATION` 구조를 기본으로 한다.
- 링크는 텍스트 링크 중심이다.
- 아이콘 버튼보다 `ul/li` 링크가 우선이다.
- Admin 링크는 있어도 되지만 공개 방문자에게는 “Editor Only”처럼 명확히 표시한다.

### 버튼과 폼

- 기본 브라우저 버튼/인풋 느낌을 유지한다.
- 폼은 테이블 셀이나 사이드바 안에 단순하게 둔다.
- 버튼은 큰 CTA처럼 만들지 않는다. “남기기”, “저장”, “삭제”처럼 기능만 말한다.

### 목록

- 글 목록, 최근 글, 미디어 목록은 테이블 우선.
- 비어 있음, 로딩, 에러도 테이블 row나 단순 문장으로 보여준다.
- 날짜는 좁은 오른쪽 칸에 둔다.

### 글 상세

- 본문 가독성은 유지하되, 모던 블로그처럼 과하게 정제하지 않는다.
- 이미지, 코드, blockquote는 기본 테이블 안에서 단순하게 보인다.
- 글 상세 하단에는 `글 목록으로`, `Home` 같은 텍스트 링크를 둔다.

### 방명록

- 방명록은 옛 웹의 핵심 기능이다.
- 입력 폼은 사이드바, 목록은 본문 영역에 두는 현재 구조가 기준이다.
- 각 entry는 dashed border와 작은 meta 정보를 유지한다.

## 카피 톤

좋은 톤:

- 조금 장난스럽고 직접적이다.
- 자기소개는 과장된 브랜딩보다 개인 홈페이지처럼 쓴다.
- 한국어 문장 안에 옛 웹 영어 라벨을 섞는다.
- “부담스럽게 반겨요”, “한 줄 남겨주세요”처럼 사람 냄새가 난다.

피할 톤:

- 스타트업 랜딩페이지처럼 멋있게 포장하는 문장
- “혁신적인”, “AI-powered”, “seamless experience”류의 추상 카피
- 너무 깔끔한 포트폴리오 문체
- 농담이 많아서 정보가 묻히는 문체

## CMS 데이터가 들어올 때의 규칙

PocketBase에서 불러온 데이터도 디자인 문법을 깨면 안 된다.

- 글 제목은 링크 또는 헤딩으로 단순하게 표시한다.
- 글 목록은 테이블 row로 렌더링한다.
- 방명록은 `.entry` 박스 안에 표시한다.
- CMS 연결 실패는 숨기지 말고 “CMS 연결 실패”처럼 방문자가 이해 가능한 문장으로 보여준다.
- 로딩 상태는 스피너보다 `불러오는 중...` 텍스트를 쓴다.

## 관리자 화면 예외

`admin/` 화면은 공개 사이트와 목적이 다르다.

- 관리자 화면은 편집 효율, 읽기 쉬움, 실수 방지가 우선이다.
- 공개 사이트만큼 과한 마퀴/GIF/깜빡임을 강제하지 않는다.
- 그래도 전체 프로젝트 톤과 충돌하지 않게 너무 고급 SaaS처럼 만들지는 않는다.

## 새 공개 페이지 만들 때 레시피

1. `index.html`을 먼저 보고 현재 감성을 확인한다.
2. 페이지가 복잡하면 홈의 2-column table shell을 재사용한다.
3. 페이지가 단순하면 760px one-cell table로 만든다.
4. 첫 화면 어딘가에 옛 웹 신호를 하나 이상 둔다.
   - marquee, badge, GIF, `UNDER CONSTRUCTION`, WebRing, counter, table header 등
5. 목록/데이터는 카드보다 테이블로 만든다.
6. 링크는 기본 파란 링크를 유지한다.
7. CSS를 추가해야 하면 `css/styles.css`에 작은 class로 추가한다.
8. 화면 변경 후 `npm run dev`로 실제 브라우저에서 확인한다.

## 변경 전 체크리스트

공개 UI를 바꾸기 전에 스스로 확인한다.

- 이 변경이 현재 홈보다 현대적인 랜딩페이지처럼 보이지 않는가?
- `760px` 테이블 세계관을 깨지 않는가?
- 링크, 폼, 목록이 브라우저 기본 느낌을 유지하는가?
- GIF/마퀴/배지/카운터 같은 옛 웹 신호가 남아 있는가?
- CMS에서 들어온 동적 콘텐츠도 같은 감성 안에 들어오는가?
- 사용자가 코드를 보지 않아도 화면에서 “아, 이 사이트 감성 유지됐네”라고 느낄 수 있는가?

## 의도적으로 디자인을 바꿀 때

홈페이지의 감성을 바꾸는 변경은 단순 구현이 아니라 제품 방향 변경이다.

그 경우:

1. 왜 바꾸는지 먼저 설명한다.
2. `design.md`를 함께 업데이트한다.
3. 필요하면 `STATUS.md`에도 현재 디자인 결정으로 기록한다.
4. 바뀐 화면을 브라우저에서 확인한다.
